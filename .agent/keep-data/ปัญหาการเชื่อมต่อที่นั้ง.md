# ปัญหาการเชื่อมต่อที่นั้ง

## อาการที่พบ

- หน้า `http://localhost:5173/booking/seats?roundId=28` เข้าได้ แต่เลือกที่นั่งต่อ flow ไม่ได้ตามปกติ
- เมื่อกดเลือกที่นั่งแล้วระบบไม่สามารถสร้าง `sessionToken` เพื่อส่งต่อไปหน้า `form` ได้
- ผลกระทบคือ flow `seats -> form -> insurance -> rental/ticket` ขาดตอน เพราะ state การจองไม่ถูกเก็บใน `booking_sessions`

## สาเหตุจริง

- หน้า `booking/seats.html` และหน้า booking HTML ชุดเดียวกันโหลดค่า API จากไฟล์ `websilte/src/assets/js/config.js`
- ไฟล์นี้ยัง hardcode endpoint เป็น `http://154.197.124.146:5000/api`
- แต่ตอน dev หน้าเว็บรันที่ `http://localhost:5173` และควรเรียก backend ผ่าน Vite proxy ที่ `/api`
- ทำให้ request สำคัญ เช่น
  - `GET /bus-rounds`
  - `GET /seat-bookings/round/:roundId`
  - `POST /booking-sessions`
  - `POST /seat-bookings/hold`
  ไปคนละ host กับ backend ที่ใช้งานจริง

## ผลลัพธ์ของ bug

- หน้าเลือกที่นั่งโหลดข้อมูลรอบรถหรือสถานะที่นั่งผิดที่
- การสร้างหรือ resume `booking session` ล้มเหลว
- token ไม่ถูก generate หรือถูกเก็บไม่ตรงระบบ
- หน้า `form` จึงไม่สามารถ resume ข้อมูลจาก token เดิมได้

## การแก้ไขที่ทำแล้ว

- แก้ `websilte/src/assets/js/config.js` ให้เลือก API endpoint ตาม host ปัจจุบัน
- ถ้าเปิดผ่าน Vite dev server ที่ port `5173` ให้ใช้ `/api`
- ถ้าเปิดจาก host อื่น ให้ fallback เป็น `http://<current-host>:5000/api`

## logic หลังแก้

```js
const getApiBaseUrl = () => {
    const currentHost = window.location.hostname || 'localhost';
    const currentPort = window.location.port;

    if (currentPort === '5173') {
        return '/api';
    }

    return `http://${currentHost}:5000/api`;
};
```

## สรุป

ปัญหาไม่ได้อยู่ที่ route `booking/seats` โดยตรง แต่เกิดจากหน้า seat ใช้ API base คนละตัวกับ environment ที่รันจริง ทำให้ flow การสร้าง token สำหรับเก็บข้อมูลจองขาด เมื่อแก้ API base ให้ตรง environment แล้ว หน้าเลือกที่นั่งกลับมาสร้าง session และส่งต่อไปหน้า form ได้ตามปกติ
